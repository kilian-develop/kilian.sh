## 문제를 느끼게 된 순간

프로젝트 초기에는 전형적인 레이어드 아키텍처를 사용했습니다. Controller - Service - Repository, 세 계층으로 나누고 각 계층에서 바로 아래 계층을 호출하는 구조. 처음에는 깔끔해 보였지만 프로젝트가 커지면서 문제가 보이기 시작했습니다.

Service 계층에 비즈니스 로직과 JPA 엔티티 조작 코드, 외부 API 호출 코드가 뒤섞이기 시작한 겁니다. 테스트를 작성하려면 데이터베이스와 외부 서비스를 모두 모킹해야 했고, 하나의 Service 클래스가 수백 줄을 넘기는 경우도 생겼습니다.

**"비즈니스 로직이 기술적 구현에 오염되고 있다"**는 느낌을 받았습니다. 그래서 클린 아키텍처를 적용해보기로 했습니다.

## 도메인 계층부터 설계하기

클린 아키텍처의 핵심은 **의존성의 방향**입니다. 안쪽 계층(도메인)은 바깥쪽 계층(인프라)을 절대 알지 못합니다. 도메인 모델은 순수 Java 객체로만 구성하고, Spring이나 JPA에 대한 의존성을 완전히 제거했습니다.

```java
// domain/Order.java - 순수 도메인 모델
@Builder(access = AccessLevel.PRIVATE, toBuilder = true)
public final class Order {
    private final String orderId;
    private final String customerId;
    private final OrderStatus status;
    private final List<OrderItem> items;
    private final LocalDateTime createdAt;

    public static Order create(String customerId, List<OrderItem> items) {
        if (items == null || items.isEmpty()) {
            throw new InvalidOrderException("주문 항목이 비어있습니다");
        }

        return Order.builder()
            .orderId(UUID.randomUUID().toString())
            .customerId(customerId)
            .status(OrderStatus.PENDING)
            .items(List.copyOf(items))
            .createdAt(LocalDateTime.now())
            .build();
    }

    public Order approve() {
        if (this.status != OrderStatus.PENDING) {
            throw new InvalidOrderException("대기 중인 주문만 승인할 수 있습니다");
        }
        return this.toBuilder()
            .status(OrderStatus.APPROVED)
            .build();
    }

    public Money calculateTotalPrice() {
        return items.stream()
            .map(OrderItem::calculateSubtotal)
            .reduce(Money.ZERO, Money::add);
    }
}
```

여기서 중요한 포인트가 몇 가지 있습니다. 첫째, `@Entity`나 `@Table` 같은 JPA 어노테이션이 없습니다. 둘째, 모든 필드가 `final`이고 `toBuilder()`를 통해 상태 변경 시 새 객체를 반환합니다. 셋째, `approve()` 같은 메서드에 비즈니스 규칙이 담겨있습니다.

정보전문가 패턴을 적용해서, 주문의 상태를 판단하는 로직은 주문 객체 자신이 갖고 있습니다. 외부에서 `order.getStatus()` 로 상태를 꺼내서 if문으로 검사하는 게 아니라, `order.approve()` 라고 의도를 전달하면 객체 스스로 판단합니다.

## Port & Adapter 패턴

도메인 계층이 외부 세계와 소통하려면 Port(인터페이스)가 필요합니다. 도메인이 필요한 기능을 인터페이스로 정의하고, 인프라 계층에서 이를 구현합니다.

```java
// domain/port/OrderRepository.java - 도메인이 정의하는 Port
public interface OrderRepository {
    Order findById(String orderId);
    void save(Order order);
    List<Order> findByCustomerId(String customerId);
}

// domain/port/PaymentProcessor.java
public interface PaymentProcessor {
    PaymentResult process(Order order, PaymentMethod method);
}
```

Port의 이름은 도메인 언어로 표현합니다. `JpaOrderDao`가 아니라 `OrderRepository`입니다. 도메인 입장에서는 데이터가 어디에 저장되는지 알 필요가 없습니다. 메모리든, PostgreSQL이든, MongoDB든 상관없이 "주문을 저장하고 조회하는 역할"만 정의합니다.

Adapter는 이 Port를 구체적인 기술로 구현합니다.

```java
// infrastructure/persistence/JpaOrderRepository.java
@Repository
@RequiredArgsConstructor
public class JpaOrderRepository implements OrderRepository {
    private final OrderJpaRepository jpaRepository;
    private final OrderMapper mapper;

    @Override
    public Order findById(String orderId) {
        return jpaRepository.findById(orderId)
            .map(mapper::toDomain)
            .orElseThrow(() ->
                new OrderNotFoundException("주문을 찾을 수 없습니다: " + orderId));
    }

    @Override
    public void save(Order order) {
        OrderEntity entity = mapper.toEntity(order);
        jpaRepository.save(entity);
    }

    @Override
    public List<Order> findByCustomerId(String customerId) {
        return jpaRepository.findByCustomerId(customerId).stream()
            .map(mapper::toDomain)
            .toList();
    }
}
```

여기서 `OrderMapper`가 도메인 모델과 JPA 엔티티 사이를 변환합니다. 매핑 코드가 번거로울 수 있지만, 이 분리 덕분에 도메인 모델은 JPA 프록시나 지연 로딩 같은 기술적 제약에서 자유로워집니다.

## 패키지 구조

패키지 구조를 어떻게 잡을지도 많이 고민했습니다. 결국 도메인 기능 중심으로 나누고, 각 도메인 안에서 계층을 분리하는 방식을 택했습니다.

```
com.example.shop
├── order
│   ├── domain
│   │   ├── Order.java
│   │   ├── OrderItem.java
│   │   ├── OrderStatus.java
│   │   └── port
│   │       ├── OrderRepository.java
│   │       └── PaymentProcessor.java
│   ├── service
│   │   ├── CreateOrderService.java
│   │   ├── ApproveOrderService.java
│   │   └── command
│   │       └── CreateOrderCommand.java
│   ├── infrastructure
│   │   ├── persistence
│   │   │   ├── JpaOrderRepository.java
│   │   │   ├── OrderEntity.java
│   │   │   └── OrderJpaRepository.java
│   │   └── payment
│   │       └── StripePaymentProcessor.java
│   └── presentation
│       ├── OrderController.java
│       └── dto
│           ├── CreateOrderRequest.java
│           └── OrderResponse.java
└── customer
    ├── domain
    ├── service
    ├── infrastructure
    └── presentation
```

이 구조의 장점은 `order` 패키지만 보면 주문과 관련된 모든 코드를 파악할 수 있다는 것입니다. 그리고 `domain` 패키지에는 Spring 의존성이 전혀 없으므로, 순수 단위 테스트로 비즈니스 로직을 검증할 수 있습니다.

## 결과와 느낀 점

클린 아키텍처를 적용한 뒤 가장 달라진 점은 **테스트 작성이 편해졌다**는 것입니다. 도메인 로직은 외부 의존성 없이 테스트할 수 있고, 서비스 계층은 Port를 모킹하면 됩니다. 데이터베이스를 띄우거나 외부 API를 호출하지 않아도 핵심 비즈니스 로직을 검증할 수 있게 되었습니다.

물론 트레이드오프도 있었습니다. 매핑 코드가 늘어나고, 작은 기능 하나를 추가할 때도 여러 계층을 거쳐야 합니다. 초기에는 과도하게 느껴질 수 있습니다. 하지만 프로젝트가 커질수록 이 구조가 주는 안정감은 확실히 체감되었습니다.

결국 아키텍처 선택은 **프로젝트의 복잡도와 팀의 상황에 맞춰야 한다**고 생각합니다. 모든 프로젝트에 클린 아키텍처가 필요하진 않습니다. 하지만 도메인 로직이 복잡하고 장기적으로 유지보수해야 하는 프로젝트라면, 이 투자는 충분히 가치가 있었습니다.
