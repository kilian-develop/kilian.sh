## 왜 블로그를 시작하게 되었나

개발을 하면서 수많은 기술적 결정을 내리게 됩니다. 어떤 아키텍처를 선택할지, 왜 이 패턴을 쓰는 게 나은지, 이 문제를 어떻게 풀어야 할지. 매일매일 고민하지만, 시간이 지나면 그때의 맥락과 판단 기준이 흐릿해지는 경험을 반복했습니다.

회의록처럼 결론만 남기는 게 아니라, **고민의 과정 자체를 기록해두고 싶다**는 생각이 들었습니다. 어떤 선택지가 있었고, 왜 그중에서 이걸 골랐는지. 그 흐름을 나중에 다시 꺼내볼 수 있다면, 같은 실수를 줄이고 더 나은 결정을 내릴 수 있을 거라 생각했습니다.

## 어떤 글을 쓸 건가

이 블로그에서는 주로 이런 이야기를 다루려고 합니다.

- **아키텍처와 설계 고민**: 클린 아키텍처, DDD, 포트앤어댑터 패턴 등을 실무에 적용하면서 겪은 시행착오
- **Spring/Java 개발 이야기**: 일상적인 백엔드 개발에서 마주친 문제와 해결 과정
- **테스트와 코드 품질**: 테스트 가능한 코드를 만들기 위한 구조적 고민
- **프론트엔드 탐험**: React Router, TypeScript 등 프론트엔드 기술도 가끔 다룰 예정

기술 자랑보다는 **"왜 그렇게 했는가"**에 초점을 맞추고 싶습니다.

## 이 블로그의 기술 스택

이 블로그 자체도 하나의 사이드 프로젝트입니다. React Router v7의 프리렌더 기능을 활용해서 정적 사이트로 만들었습니다.

```tsx
// routes.ts - 라우트 설정은 이렇게 간단합니다
import { type RouteConfig, index, route } from "@react-router/dev/routes";

export default [
  index("routes/home.tsx"),
  route("blog", "routes/blog._index.tsx"),
  route("blog/:slug", "routes/blog.$slug.tsx"),
] satisfies RouteConfig;
```

MDX로 글을 작성하면 빌드 타임에 HTML로 변환되기 때문에, 별도의 CMS나 데이터베이스 없이도 빠르고 가벼운 블로그를 운영할 수 있습니다.

```bash
# 새 글을 작성하는 과정
# 1. content/posts/ 디렉토리에 MDX 파일 생성
# 2. frontmatter에 메타데이터 작성
# 3. 본문을 마크다운으로 작성
# 4. 빌드하면 끝
npm run build
```

## 앞으로의 다짐

완벽한 글을 쓰려고 하기보다는, 겪은 고민을 솔직하게 풀어내는 것부터 시작하려 합니다. 불완전한 시도도 누군가에게는 공감이 될 수 있다고 믿으면서.

첫 글은 여기서 마무리하겠습니다. 앞으로 꾸준히 기록해 나가겠습니다.
