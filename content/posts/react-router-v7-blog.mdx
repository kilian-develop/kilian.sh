## 왜 React Router v7을 선택했나

블로그를 만들겠다고 마음먹었을 때, 프레임워크 선택이 첫 번째 고민이었습니다. Next.js, Astro, Gatsby 등 좋은 선택지가 많았지만, 결국 React Router v7을 선택했습니다.

이유는 단순했습니다. **Remix에서 이어진 loader/action 패턴이 직관적이었고**, v7부터 프레임워크 모드를 지원하면서 별도 설치 없이 SSG까지 가능해졌기 때문입니다. 이미 React Router의 라우팅 모델에 익숙했기 때문에 학습 비용도 낮았습니다.

다만 처음부터 순탄하지만은 않았습니다. 문서가 아직 완벽하지 않은 부분도 있었고, MDX 통합 과정에서 예상치 못한 문제를 만나기도 했습니다.

## 라우트 설계

React Router v7의 Framework Mode에서는 `routes.ts` 파일에서 라우트를 선언적으로 정의합니다. 블로그에 필요한 라우트는 크게 세 가지였습니다.

```typescript
// app/routes.ts
import { type RouteConfig, index, route } from "@react-router/dev/routes";

export default [
  index("routes/home.tsx"),
  route("blog", "routes/blog._index.tsx"),
  route("blog/:slug", "routes/blog.$slug.tsx"),
] satisfies RouteConfig;
```

파일 기반 라우팅도 지원하지만, 명시적으로 정의하는 방식이 프로젝트 구조를 한눈에 파악하기 좋았습니다. `blog/:slug` 패턴으로 동적 라우트를 잡아두면, `content/posts/` 디렉토리의 MDX 파일 이름이 곧 URL 슬러그가 됩니다.

## Loader 패턴으로 데이터 불러오기

React Router v7의 핵심 중 하나는 loader 패턴입니다. 서버 사이드(또는 빌드 타임)에서 데이터를 미리 가져와 컴포넌트에 전달합니다. 블로그 목록 페이지의 loader는 이렇게 생겼습니다.

```tsx
// app/routes/blog._index.tsx
import type { Route } from "./+types/blog._index";
import { getAllPosts } from "~/lib/posts.server";

export async function loader(_: Route.LoaderArgs) {
  const posts = await getAllPosts();
  return { posts };
}

export default function BlogIndex({ loaderData }: Route.ComponentProps) {
  const { posts } = loaderData;

  return (
    <div>
      {posts.map((post) => (
        <PostCard key={post.slug} post={post} />
      ))}
    </div>
  );
}
```

`posts.server.ts`라는 이름에서 `.server` 접미사가 중요합니다. 이 파일은 서버 환경에서만 실행되므로 `fs`나 `path` 같은 Node.js 모듈을 자유롭게 사용할 수 있습니다. 클라이언트 번들에는 포함되지 않습니다.

## MDX 통합 과정

MDX를 React Router v7에 통합하는 게 가장 많은 시행착오를 겪은 부분이었습니다. 처음에는 런타임에서 MDX를 파싱하려 했지만, 빌드 타임에 처리하는 게 훨씬 효율적이었습니다.

```typescript
// vite.config.ts
import mdx from "@mdx-js/rollup";
import remarkFrontmatter from "remark-frontmatter";
import remarkMdxFrontmatter from "remark-mdx-frontmatter";
import rehypeSlug from "rehype-slug";
import rehypeAutolinkHeadings from "rehype-autolink-headings";

export default defineConfig({
  plugins: [
    {
      enforce: "pre",
      ...mdx({
        remarkPlugins: [remarkFrontmatter, remarkMdxFrontmatter],
        rehypePlugins: [rehypeSlug, rehypeAutolinkHeadings],
      }),
    },
    tailwindcss(),
    reactRouter(),
    tsconfigPaths(),
  ],
});
```

여기서 핵심은 `enforce: "pre"` 설정입니다. MDX 플러그인이 React Router 플러그인보다 먼저 실행되어야 `.mdx` 파일이 제대로 변환됩니다. 이걸 빠뜨리면 빌드 에러가 나는데, 처음에는 원인을 찾느라 꽤 헤맸습니다.

그리고 개별 포스트 페이지에서는 Vite의 `import.meta.glob`을 활용해 모든 MDX 파일을 빌드 타임에 가져옵니다.

```tsx
// app/routes/blog.$slug.tsx
const mdxModules = import.meta.glob("../../content/posts/*.mdx", {
  eager: true,
}) as Record<string, { default: React.ComponentType }>;

export default function BlogPost({ loaderData }: Route.ComponentProps) {
  const { post } = loaderData;

  const mdxKey = Object.keys(mdxModules).find((key) =>
    key.endsWith(`/${post.slug}.mdx`)
  );
  const MdxContent = mdxKey ? mdxModules[mdxKey].default : null;

  return (
    <article className="prose-blog">
      {MdxContent && <MdxContent />}
    </article>
  );
}
```

loader에서 React 컴포넌트를 직렬화할 수 없기 때문에, loader는 메타데이터만 반환하고 MDX 컴포넌트는 별도로 임포트하는 구조입니다. 이 분리가 처음에는 어색했지만, 관심사 분리 관점에서 오히려 깔끔하다고 느꼈습니다.

## 배운 점

React Router v7로 블로그를 만들면서 가장 크게 느낀 점은, **프레임워크의 제약이 오히려 좋은 구조를 만들어준다**는 것입니다. loader에서 데이터를 가져오고 컴포넌트에서 렌더링하는 패턴이 강제되면서, 데이터 흐름이 자연스럽게 단방향으로 정리되었습니다.

아직 개선할 부분이 많습니다. 신택스 하이라이팅, 목차 자동 생성, 검색 기능 등을 하나씩 추가해 나갈 예정입니다. 그 과정도 블로그에 기록할 생각입니다.
